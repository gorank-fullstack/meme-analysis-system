- 统一第三方 API 数据结构与差异

## 📂 src/utils/api

### 职责定位

`api` 目录用于：

- 封装 Redis / API / 缓存层的读取逻辑
- 统一返回结构，避免业务层直接依赖第三方差异
- 提供“已处理过”的稳定数据入口

**原则：**
- 不直接参与 UI 或业务判断
- 不包含评分、排序等算法逻辑
- 只做：获取、整理、统一

### 示例功能类型

- 从 Redis / API 读取 Token 列表
- 将多来源数据整合为 `uniform-data` 结构
- 屏蔽不同链（EVM / SPL）字段差异

## 📂 src/utils/format

## 📂 src/utils/sort/uniform-data

> 🔥 **Token 热度评分与排行榜系统核心**

这是整个系统中**最核心、最不可替代的算法模块**。

---

### 🧠 模块定位

该目录用于实现：

- Token 热度评分（Hot Score）
- 多时间维度排行榜（5m / 1h / 6h / 24h）
- 防刷榜、反作弊、风险降权
- 热度数据的裁剪、重建与可解释排序

这是一个**可调参、可解释、可扩展**的评分系统。

---

### 📄 核心文件

#### 1️⃣ `sort-hot_rank.ts`-- 代码省略，不开源

🔥 **热度计算与排序引擎（主流程）**

主要职责：

- 热度 Map 的生命周期管理
- Token 活跃交易额计算
- 排行榜构建与裁剪
- 多时间维度同步更新

##### 核心流程（顺序不可随意调整）

1. **`prune_And_ResetZero_HotMap`**
   - 清理超时 Token
   - 管理命中次数与清零周期

2. **`filterHighRiskToken`（可选）**
   - EVM / SOL 分链风险过滤
   - 高风险 Token 直接剔除

3. **`calculateActiveTradingVolume`**
   - 核心评分计算阶段
   - 计算 `lv_0_hot_vol[qt]`
   - 引入多维度权重与修正系数

4. **`rebuildSortedList`**
   - 支持任意字段排序
   - 支持 number / string / 时间字段

5. **`clipHotMap`**
   - 超量裁剪
   - 优先保留最近命中的 Token

---

#### 2️⃣ `sort-hot_rule.ts`-- 代码省略，不开源

🧮 **评分规则与权重函数定义**

用于集中管理所有“评分因子”，避免散落在主流程中。

##### 已实现规则类型

所有规则均为**函数化、可替换、可调参**设计。

---
### 🧩 评分体系设计说明

- **Lv0 / Lv1 / Lv2 分级结构**
- 所有中间计算结果可追溯
- 支持输出评分公式字符串（仅开发期使用）
- 不依赖数据库、不产生副作用

---

### ⚠️ 重要设计约定

- **EVM 与 SPL 安全模型严禁强行统一**
- 排序模块不关心 UI、API、存储
- 所有评分只依赖 `uniform-data` 输入
- 本模块属于系统“智力资产”，修改需极度谨慎

## 📂 src/utils/task

### 职责定位

`task` 目录用于存放**定时任务相关的通用辅助工具**，主要解决：

* NestJS `@Cron` 在高频调度下的**重复执行问题**
* 秒级抖动、服务重启、调度延迟带来的**多次触发风险**
* Cron 表达式语义复杂、可读性差的问题

该目录的目标是：

> **让定时任务“只在应该执行的那一分钟，执行一次”。**

---

## 📄 time-task-helper.ts

### 🧠 模块定位

`TimeTaskHelper` 是一个 **Cron 容错与幂等执行辅助器**，用于配合 NestJS 的 `@Cron` 装饰器使用。

它在不破坏 Cron 原有调度机制的前提下，实现：

* ✅ **每分钟仅执行一次**
* ✅ **支持秒级容错窗口**
* ✅ **支持多分钟 / 间隔分钟**
* ✅ **支持固定小时 / 间隔小时**
* ✅ **防止服务抖动、重复触发**

---

### 🎯 设计目标

1. **不依赖 Redis / DB**
2. **零侵入业务逻辑**
3. **只负责“是否该执行”**
4. **Cron 表达式 → 可读参数结构**
5. **逻辑可审计、可推导**

---

## 📌 总体设计原则

- utils ≠ helper
- sort ≠ 简单排序
- 所有复杂度集中在算法层
- 上层（API / UI）保持尽量“简洁”
